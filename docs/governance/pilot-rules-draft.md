# Pilot Rules Draft

Date: March 23, 2026
Status: Internal planning only

## Purpose

This document defines a proposed baseline for municipal pilots before any hard product enforcement is introduced.

## Recommended default pilot profile

- One tenant organization
- One to two active production domains
- Ten to twenty-five named users
- Sixty to ninety day pilot window
- Business-hours support only unless a separate commitment is approved

These defaults are intentionally conservative. The goal is to learn quickly without allowing the pilot to silently become an unrestricted production rollout.

## Core pilot rules

### Scope
- Pilot access is limited to the approved municipality or operating entity
- Use is limited to the approved operational workflows documented during onboarding
- Public claims, press references, or branding use require explicit approval

### Domain limits
- Each pilot should include an explicit cap on active domains
- The default starting cap should be `1` or `2` production domains
- Additional domains require review and approval before activation

### User limits
- Each pilot should include an explicit cap on named tenant users
- The default starting cap should be `10` to `25` named users
- Additional users require review and approval before activation once pricing is live

### Term
- Every pilot should have a start date and target end date
- The default pilot window should be `60` to `90` days
- Extension decisions should be explicit, documented, and auditable

### Support and service expectations
- No implied 24x7 support
- No guaranteed response or repair outcome through the platform
- Pilot support should be framed as onboarding, issue triage, and bounded operational support

### Permitted use
- Use is restricted to the named tenant and approved participant accounts
- Access should not be shared broadly outside authorized users
- Resale, white-labeling, and unrelated third-party operational use should be prohibited

## Definitions that must be locked

### What counts as a domain
Recommended starting definition:
- A tenant-enabled operational module or report domain activated for real tenant use

### What counts as a user
Recommended starting definition:
- A named account with active access to the tenant workspace or tenant map/reporting environment

### What happens after pilot expiry
Options to evaluate:
- read-only access for a short transition window
- export-only transition mode
- access pause until paid conversion or approved extension

## Product vs process boundary

### Contract/process only for now
- pricing terms
- acceptance language
- pilot duration
- support scope
- branding/announcement approval
- exception authority

### Future in-product controls
- warning banners as pilots approach caps or expiry
- soft cap alerts for users and domains
- confirmation checkpoints before billable expansion
- audit logs for approvals and acknowledgements
- eventual hard enforcement if policy requires it

## Operating guidelines

- Keep pilot scope narrow enough to measure adoption and workflow value clearly
- Avoid adding domains or users without a named owner approving the expansion
- Treat exceptions as explicit decisions, not quiet defaults
- Document any granted exception with date, approver, and reason

## Immediate recommendation

Adopt a default pilot rule set of:
- max `2` active domains
- max `25` named users
- `90` day pilot term

Allow deviations only through documented approval while commercialization rules are still being finalized.
